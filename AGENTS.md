# AGENTS.md

本文件面向在本仓库内工作的智能编码代理。目标是把规则压缩成开发时容易持续遵守的执行清单；当规则与现有业务行为冲突时，优先保证现有业务行为不被破坏。

## 0. 最高优先级硬规则

每次改代码前先读本节；交付前再按本节回看一遍。

- 回复必须使用中文简体。
- 务必按照分层规范 `Controller - Facade（业务实现类） - Service(数据操作接口类) - ServiceImpl(数据操作实现类)` 开发。
- 未经用户明确要求，禁止主动执行编译、打包、格式化命令，包括 `mvn compile`、`mvn test`、`mvn package`、带 `-am` 的模块命令和全量命令。
- 验证测试统一生成可导入postman的测试用例文件即可，请勿增加junit测试代码
- 关键业务判断、状态流转、异步、批量、延迟消息、事务边界等逻辑必须加有信息量的 `//` 注释或 Javadoc，说明业务意图；不要写“设置值/调用方法”这类废话注释。
- 是否遵循 `默认工作流程`、`数据类型与转换`、`枚举、常量与魔法值`、`异常与日志`要求
- 未经用户明确要求，禁止删除、弱化或顺手清理用户已有注释、业务说明和占位提示。

## 2. 默认工作流程

### 2.1 修改前

- 先看现有目录结构、命名、分层、相邻代码风格。
- 先查是否已有 service、枚举、常量、convert、XML、工具方法可复用。
- 判断查询是否单表：单表走 MP Wrapper；跨表/聚合/复杂关联才考虑 XML。
- 判断校验是否已由 BO 上的 `jakarta.validation` 注解表达；已表达的结构性校验不要在 controller / feign 实现层重复手写。
- 不为了“更抽象”新增无意义封装；不主动升级依赖、重构大面积代码、修改基础设施配置。

### 2.2 修改中

- 优先做最小必要改动，避免扩大 diff。
- 场景编排、权限判断、状态流转、字段补充放 facade。
- 集合判空统一使用 `CollectionUtil.isEmpty` / `CollectionUtil.isNotEmpty`；字符串判空统一使用 `StrUtil.isBlank` / `StrUtil.isNotBlank`；等值判断优先使用 `Objects.equals`。
- mapper 调用、MP 查询、批量数据处理、可复用能力放 service。
- controller 只做参数接收、校验、调用 facade、返回 `R.ok(...)`。
- 不在 `for` 循环内逐条查询或更新；优先批量查询、批量保存、批量更新、批量删除。
- 不要把只在一个 facade 内使用的逻辑强行下沉到 service。
- 若只是简单转发 `list`、`saveBatch`、`updateBatchById` 等内置能力，不要额外包一层同义方法。

### 2.3 交付前

必须自检以下事项，发现问题先修正再交付：

- 是否误用 XML 处理单表查询或单表分页。
- 集合、字符串、等值判断是否使用指定工具类。
- 关键业务逻辑是否补了有信息量的 `//` 注释或 Javadoc。
- 枚举字段是否只传 code，是否避免了重复 code-to-desc 私有逻辑。
- 写操作入口是否需要 `@Transactional(rollbackFor = Exception.class)`。
- 是否存在注解校验和手写同义空校验重复。
- 异常语义是否过泛，日志上下文是否足够定位问题。
- 是否误删了用户已有注释或做了无关格式化。
- 是否在用户未要求时执行了编译、测试、打包、格式化命令。

## 3. 分层规范

### 3.1 facade

- 承载当前业务场景的流程编排、权限判断、状态流转、字段补充。
- 可以保留仅服务当前场景的私有步骤方法。
- 复杂业务步骤前补充业务意图注释。

### 3.2 service

- 负责 mapper 调用、MyBatis-Plus 查询、批量数据处理、可复用公共能力。
- 只有明确存在跨场景复用价值时才抽到 service。
- 调用继承自 `ServiceImpl` / `MpBaseServiceImpl` 的批量写方法时，按最高优先级硬规则补事务。

### 3.3 mapper / XML

- 单表查询必须优先 MyBatis-Plus Wrapper，单表分页也不例外。
- 跨表字段、聚合字段、复杂关联字段优先使用 XML 关联查询。
- 不要在通用单表查询里混入复杂关联字段。

### 3.4 controller / feign / job / mq consumer

- 入口层参数结构校验优先使用 `@Valid` / `@Validated`。
- 不在入口层写数据库查询、复杂权限判断或流程编排。
- 新增接口时补充与现有代码一致的 `@Operation`、必要的 `@Schema`。

## 4. 数据、类型与转换

- BO 表示入参对象，VO 表示出参对象，DTO 表示传输对象；不要直接把 entity 暴露给 controller。
- 基础字段映射优先放到 `convert` 包下的 MapStruct 接口。
- 特殊字段、描述字段、权限过滤字段在业务层补充设置。
- MapStruct 无法自动映射的字段，不需要为了消除警告而机械添加 `ignore`。
- ID 类型遵循现有模型；如果 VO 已使用 `String`，不要随意改为 `Long`。
- Map 类型变量名必须表达映射关系，例如 `projectToStaffMap`、`stageToTaskListMap`。
- 删除统一按逻辑删除语义处理，遵循现有 `remove` / `isDeleted` 规则。

## 5. 枚举、常量与魔法值

- 业务状态值优先定义到枚举类，枚举名以 `Enum` 结尾。
- 处理字段取值、分支判断、入参校验前，先查字段是否已有枚举、常量或转换逻辑。
- 若字段已有枚举，必须复用，不要在 facade/service 中重复写 code-to-desc 私有逻辑。
- 枚举类可补充 `getDescByCode` 一类静态能力。
- 常量统一放 `constants` 包，禁止新增难理解的裸字面量。

## 6. 异常与日志

- 统一使用异常枚举抛业务异常，例如 `ExceptionEnum.PROJECT_NOT_EXIST.error()`。
- 新增业务错误优先扩展现有 `ExceptionEnum`；不要随意抛裸 `RuntimeException`。
- 业务失败不能默认落到 `REQ_PARAM_ILLEGAL`、`REQ_ILLEGAL`、`PLEASE_TRY_AGAIN_LATER` 等泛化异常。
- 只有系统级错误、下游调用失败、数据落库失败、批处理失败等需要排障介入的异常才使用 `log.error`。
- 参数校验、业务校验、状态拦截等预期内失败按需使用 `log.info` / `log.warn`，或直接抛业务异常。
- 日志必须带可定位上下文；批量接口至少记录主键集合、目标状态、失败原因或下游响应。
- 不要打印敏感信息、整段大对象或无上限列表。

## 7. 命名、导入与可读性

- 类名使用业务名词全称，避免模糊缩写。
- 方法名使用“动作 + 对象 + 返回语义”风格，如 `getProjectInfoList`、`getAppDetailVoById`。
- 避免新增无边界的 `*` 导入；除非正在修改相关行，不为旧代码的 `*` 导入做无关改动。
- Java 类型可通过 import 表达时，不要新增 `java.util.ArrayList`、`java.util.Comparator` 这类全限定写法；命名冲突无法避免时除外。
- 保持现有 Java 风格：4 空格缩进，花括号风格跟随当前文件。
- 不主动做纯格式化改动。
- 优先使用卫语句减少嵌套；复杂判断可拆为私有方法、`switch`、枚举映射或策略方式。
- 对模板复制、结构克隆等可扩展流程，不要只为当前一个步骤预留单个布尔开关；优先使用能表达多个步骤开关的配置对象或枚举。

## 8. 异步、批量与入参设计

- 新增或修改异步/批量接口时，先确认“单条数据的业务语义”和“批量入参结构”是否一致。
- 若一条记录对应一个独立状态、结果或动作，入参必须设计成明细项列表，不能偷懒设计成“ID 列表 + 单一状态”。
- 批量操作失败时输出主键集合、业务 ID、阶段信息等排障上下文。
